# CAN 协议全解：分层模型、四大帧、错误与过载机制梳理

## CAN 协议与 OSI 七层模型的关系

OSI（Open Systems Interconnection，开放式系统互联参考模型）是国际标准化组织（ISO）制定的一套通信分层标准。它的核心思想是：**把完整的数据通信流程拆分成 7 个独立层级，每层只干自己分内的事，上下层之间仅通过标准接口交互**。这样一来，不同厂商的硬件和软件只要遵守同一套分层规范，就能实现互联互通。

OSI 七层模型自上而下依次为：

| 层级 | 名称              | 英文原名           |
|------|-------------------|--------------------|
| 7    | 应用层            | Application Layer  |
| 6    | 表示层            | Presentation Layer |
| 5    | 会话层            | Session Layer      |
| 4    | 传输层            | Transport Layer    |
| 3    | 网络层            | Network Layer      |
| 2    | 数据链路层        | Data Link Layer    |
| 1    | 物理层            | Physical Layer     |

![image-20260705133616288](./assets/image-20260705133616288.png)

根据 ISO 11898 系列标准（CAN 2.0A/B、CAN FD），CAN 协议本身**仅定义了第 1 层（物理层）和第 2 层（数据链路层）**。但在实际车载诊断场景中，UDS 诊断不能直接跑在裸 CAN 上，还需要在上层叠加 ISO 15765-2（DoCAN）来封装分段重组、流控等机制。因此，一套完整的 **UDS on CAN 通信栈**，自上而下分为如下四层：

| OSI 层              | 对应标准             | 作用                                                           |
| ------------------- | -------------------- | -------------------------------------------------------------- |
| 7 — 应用层          | ISO 14229（UDS）     | UDS 诊断服务：读故障码、刷写、读数据流等业务指令               |
| 4 — 传输层 + 3 — 网络层 | ISO 15765-2（DoCAN）| 封装分段重组、流控、单帧/多帧传输，弥补单条 CAN 帧有效载荷短的缺陷 |
| 2 — 数据链路层      | ISO 11898            | 总线仲裁、帧封装、CRC 校验、错误处理、报文过滤等链路通信机制   |
| 1 — 物理层          | ISO 11898-2          | 差分总线电平、CAN 收发器、终端电阻、总线布线规范，完成电平信号收发 |

---

## 数据链路层

数据链路层介于硬件和上层应用之间，是 CAN 协议的核心。它又进一步拆分为两个子层：**MAC 子层（Media Access Control，媒介访问控制）** 和 **LLC 子层（Logical Link Control，逻辑链路控制）**。

两个子层的分工各有侧重：

- **LLC 子层**：作为上层（DoCAN / 应用）与 MAC 之间的桥梁，负责帧类型发起、ID 管理、报文过滤等逻辑控制；
- **MAC 子层**：直接操作总线比特流，负责帧的物理组装、总线传输、自动重发等底层操作。

![数据链路层的MAC和LLC子层](./assets/image-20260705020226775.png)

在日常车载诊断工作中，**99% 的情况下我们看到的数据帧，就是在第 2 层干活儿的**。因此，吃透数据链路层的仲裁机制、帧格式、错误处理等内容，对于打好 CAN 总线基础非常关键。

---

## 帧的种类

ISO 11898 定义了 **4 种帧类型**，外加一个用于帧间分隔的 **帧间隔**：

| 帧类型           | 含义                                                     |
| ---------------- | -------------------------------------------------------- |
| 🟢 **数据帧**     | 发送单元向接收单元发送有效数据                           |
| 🔵 **遥控帧**（远程帧） | 接收单元向拥有相同 ID 的发送单元请求数据             |
| 🔴 **错误帧**     | 节点检测到总线传输故障时，主动发出错误通知               |
| 🟡 **过载帧**     | 接收单元内部缓存已满、来不及处理报文时，通知发送方"尚未准备好接收" |
| ⚪ **帧间隔**     | 分隔前后相邻的数据帧或遥控帧                             |

其中，**数据帧和遥控帧**是最常打交道的两类；**错误帧和过载帧**由 CAN 控制器硬件自动处理，平时我们很少直接感知到它们，但理解其原理对于排查总线故障至关重要。

---

## 标准帧与扩展帧

CAN 2.0 规范由 Bosch 于 1991 年发布，最初规定标识符（ID）为 **11 位**，也就是所谓的**标准帧**。后来 ID 资源不够用了，Bosch 又在 CAN 2.0B 中把 ID 扩展到了 **29 位**，这就是**扩展帧**。

日常使用 Vector 等工具时，看到 `Standard` 就知道是标准帧，看到 `Extended` 就知道是扩展帧。

![标准帧与扩展帧](./assets/image-20260705023144988.png)

### 标准帧 ID 的范围

标准帧 ID 范围是 **`0x000 ~ 0x7FF`**。为什么最大值不是 `0xFFF`？

> 因为 11 位二进制数的最大值 = 2¹¹ − 1 = 2047，换算为十六进制就是 `0x7FF`。

<img src="./assets/image-20260705040108843.png" alt="ID范围计算" style="zoom:50%;" />

### 为什么诊断报文都是 `0x7xx` 开头？

在车载诊断中，物理寻址和功能寻址都以 `0x7xx` 开头，而不是 `0x1xx`、`0x2xx`、`0x3xx`。原因很简单：

**CAN 总线的仲裁规则是——ID 越小，优先级越高。** 把 `0x000 ~ 0x6FF` 留给更重要的业务报文（动力系统、底盘安全等），诊断报文统一使用 `0x7xx` 段，可以确保诊断通信不会抢占高优先级业务报文的带宽。

## 帧格式详解

数据帧和遥控帧都分别有**标准格式**和**扩展格式**两种，但它们在结构上有一些差异。下面逐一拆解。

### 5.1 数据帧的标准格式与扩展格式

数据帧由 **7 个段**组成：

```
帧起始 → 仲裁段 → 控制段 → 数据段 → CRC 段 → ACK 段 → 帧结束
```

![数据帧结构](./assets/image-20260705031057138.png)

在深入细节之前，先回顾一下上一篇学过的**逻辑电平**：

| 电平 | 含义     |
|------|----------|
| 显性 | 逻辑 `0` |
| 隐性 | 逻辑 `1` |

CAN 总线通过"线与"机制实现仲裁：显性电平会覆盖隐性电平。

---

#### ① 帧起始（SOF，Start of Frame）

一帧的开始标志。发送方发出**1 位显性电平 `0`**，打破总线空闲状态，告诉总线上所有节点："我要开始发数据了！"

SOF 之后紧接着就是 ID 字段（标准帧 11 位，扩展帧 29 位）。

---

#### ② 仲裁段

仲裁段的结构在标准帧和扩展帧中有所不同：

| 格式   | 组成                                              |
|--------|---------------------------------------------------|
| 标准帧 | 11 位 ID + 1 位 RTR                               |
| 扩展帧 | 11 位基础 ID + 1 位 SRR + 1 位 IDE + 18 位扩展 ID |

几个关键位的含义：

| 位    | 全称                          | 含义                                                                 |
|-------|-------------------------------|----------------------------------------------------------------------|
| RTR   | Remote Transmission Request   | 远程请求位。**数据帧 RTR = 0**，遥控帧 RTR = 1                       |
| SRR   | Substitute Remote Request     | 替代远程请求位。协议升级时留下的占位，**始终为 1**（隐性）            |
| IDE   | Identifier Extension          | 扩展标志位。**标准帧 IDE = 0**，扩展帧 IDE = 1                       |

![仲裁段结构](./assets/image-20260705030355553.png)

> **为什么要区分标准帧和扩展帧？**
>
> 因为标准帧和扩展帧可能共存于同一总线。节点通过 IDE 位来识别当前帧的 ID 是 11 位还是 29 位，从而正确解析后面的字段。

---

#### ③ 控制段

控制段包含保留位、IDE 位（仅扩展帧需要再次出现）和 DLC。

| 字段   | 位数 | 说明                                                              |
|--------|------|-------------------------------------------------------------------|
| IDE    | 1    | 扩展帧中再次出现（标准帧的 IDE 已在仲裁段体现）                    |
| r0/r1  | 1~2  | 保留位（Reserved），为协议未来升级预留。**必须以显性电平（0）发送** |
| DLC    | 4    | Data Length Code，指示**数据段包含多少个字节**                    |

![控制段结构](./assets/image-20260705031348818.png)

---

#### ④ 数据段（Data Field）

数据段内容长度由 DLC 决定，**0 ~ 8 字节**。CAN 2.0 标准帧/扩展帧最大有效载荷均为 8 字节（CAN FD 之后扩展到了最多 64 字节，后续文章再聊）。

**DLC 与实际数据字节数的对应关系：**

![DLC与数据字节数的对应关系](./assets/image-20260705032804164.png)

> 💡 DLC 的值可以超过 8（CAN FD），但在经典 CAN 中，DLC > 8 视为无效，按 8 字节处理。

数据段在标准帧和扩展帧中结构完全相同：

![数据段结构](./assets/image-20260705033010328.png)

---

#### ⑤ CRC 段（Cyclic Redundancy Check）

用于校验数据传输是否出现错误。标准帧和扩展帧格式相同。

| 字段         | 位数 | 说明                                             |
|--------------|------|--------------------------------------------------|
| CRC 序列     | 15   | CRC 校验值，由发送方计算并填入                   |
| CRC 界定符    | 1    | 固定为**隐性 `1`**，用于释放总线、给应答留出时间 |

![CRC段结构](./assets/image-20260705033140525.png)

> 💡 **CRC 校验算法是什么？**
>
> CAN 协议使用的 CRC 多项式为：**x¹⁵ + x¹⁴ + x¹⁰ + x⁸ + x⁷ + x⁴ + x³ + 1**（CRC-15）。
>
> 发送方按位流计算 CRC 值并填入 CRC 段；接收方用同样的多项式重新计算，若结果不一致，说明数据在传输中受损，触发**CRC 错误**。

---

#### ⑥ ACK 段（Acknowledgment，应答段）

用于确认数据是否被至少一个接收方正确接收。标准帧和扩展帧格式相同。

| 字段        | 位数 | 说明                                               |
|-------------|------|----------------------------------------------------|
| ACK 槽      | 1    | 发送方发隐性 `1`，**正确接收的节点回发显性 `0` 拉低总线** |
| ACK 界定符   | 1    | 固定为隐性 `1`，释放总线                            |

**ACK 的工作流程：**

1. 发送方在 ACK 槽发送**隐性 `1`**；
2. 所有接收到正确 CRC 校验的节点，会在 ACK 槽回应**显性 `0`** 拉低总线；
3. 发送方在 ACK 槽位置采样总线电平：
   - ✅ 读到了 `0` → **至少有一个节点正确接收**，传输成功。
   - ❌ 读到了 `1` → **无应答（ACK Error）**，触发错误处理机制。

![ACK段结构](./assets/image-20260705034214936.png)

---

#### ⑦ 帧结束（EOF，End of Frame）

**连续发送 7 位隐性 `1`**，标志本帧数据传输完毕。有始有终，SOF 用 `0` 开头，EOF 用 7 个连续的 `1` 收尾。

![帧结束结构](./assets/image-20260705033531246.png)

---

### 5.2 遥控帧的标准格式与扩展格式

遥控帧在实际应用中**极少使用**，它的作用仅仅是：**请求拥有相同 ID 的节点发送数据**。

遥控帧的结构与数据帧几乎一致，但**没有数据段**，只有 6 个部分：

```
帧起始 → 仲裁段 → 控制段 → CRC 段 → ACK 段 → 帧结束
```

![遥控帧结构](./assets/image-20260705035604002.png)

最关键的区别在于**仲裁段中的 RTR 位**：遥控帧的 **RTR = 1（隐性）**，而数据帧的 **RTR = 0（显性）**。由于显性电平会覆盖隐性电平，如果同一 ID 的数据帧和遥控帧同时发送，**数据帧会因仲裁胜出而优先发送**。

---

### 5.3 数据帧 vs 遥控帧：一张表搞懂

| 对比维度   | 数据帧                           | 遥控帧（远程帧）                    |
| ---------- | -------------------------------- | ----------------------------------- |
| RTR 位     | `0`（显性）                      | `1`（隐性）                         |
| 数据段     | 存在 0 ~ 8 字节有效数据          | **无任何数据**，数据段为空           |
| DLC 含义   | 当前发送的数据字节数             | **期望获取**的数据字节数            |
| 传输方式   | 广播式                           | 请求式                              |
| 实际场景   | 绝大多数：周期信号、诊断收发、控制指令等 | 极少使用                            |
| 发送后行为 | 发送完成，无需对方回复           | 等待目标 ECU 回复**相同 ID 的数据帧** |

## 错误帧

当总线上的任一节点检测到通信错误时，会**立刻发送错误帧**，通知所有节点"本次传输出了问题"。

### 错误类型（5 种）

| 错误类型     | 触发条件                                                     |
|--------------|--------------------------------------------------------------|
| 位错误       | 发送方回读总线电平，发现与自己发出的不一致（非仲裁阶段）       |
| 填充错误     | 连续 6 个相同电平后，没有出现填充位                           |
| CRC 错误     | 接收方计算的 CRC 与发送方填入的 CRC 不一致                    |
| 格式错误     | 固定格式位（如 CRC 界定符、ACK 界定符、EOF）电平不符合规范    |
| ACK 错误     | 发送方在 ACK 槽未检测到显性应答                               |

### 错误帧的结构

错误帧由两段组成，中间间隔 **8 位隐性分隔**：

```
错误标志段（6 位） → 8 位隐性分隔 → 错误界定符（8 位隐性 1）
```

| 节点类型       | 错误标志段内容          |
|----------------|------------------------|
| 主动错误节点   | 6 位显性 `0`           |
| 被动错误节点   | 6 位隐性 `1`           |

> **主动错误**与**被动错误**是 CAN 控制器的两种错误状态，由内部的 TEC（发送错误计数器）和 REC（接收错误计数器）决定。这个会在后续文章中详细展开。

![错误帧结构](./assets/image-20260705131140697.png)

## 过载帧

当接收单元内部缓存已满、来不及处理后续报文时，会发送**过载帧**，通知发送方"我还没准备好，先别发"。

过载帧的结构与错误帧类似：

```
过载标志（6 位显性 0） → 过载界定符（8 位隐性 1）
```

- **过载标志**：在总线"线与"机制下，任一节点发出 6 个显性位即可拉低总线，打断所有待发送报文，为接收节点争取缓冲时间。
- **过载标志重叠**：多个节点同时发送过载标志时，显性位叠加，过载标志段的总长度可能**超过 6 位**，这属于正常行为。
- **过载界定符**：8 位隐性 `1`，释放总线，等待总线恢复空闲。

![过载帧结构](./assets/image-20260705131206088.png)

---

## 帧间隔

帧间隔是**数据帧和遥控帧**之间的分隔单元，位于前一帧的 EOF 之后、下一帧的 SOF 之前。

![帧间隔结构](./assets/image-20260705132110404.png)

> ⚠️ **注意**：**错误帧和过载帧不需要帧间隔**。它们紧贴在前一帧之后直接发送，结束后接自身的界定符，不需要完整的帧间隔。

---

## 小结

本文从 OSI 七层模型出发，梳理了 CAN 协议在整个通信栈中的定位，并逐一拆解了 4 种帧类型的结构与用途：

- **数据帧**是主力选手，承载了几乎所有的业务通信；
- **遥控帧**用得极少，但在特定场景下依然存在；
- **错误帧**和**过载帧**是 CAN 总线健壮性的保障机制；
- **帧间隔**承担着帧与帧之间的"呼吸"作用。

理解了帧的结构，对后续排查总线故障、分析 Trace 将大有帮助。
